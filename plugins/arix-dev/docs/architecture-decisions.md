# arix-dev Plugin — Architecture Decisions

## AD-1: arix 전용으로 시작

**결정**: 규칙을 arix-ai 설계서에 하드코딩한다. 범용화는 후속 작업.

**근거**:
- 현재 설계 문서 4개가 구체적이고 안정적 (v3.2)
- 범용 config 시스템을 먼저 만들면 over-engineering
- 실제 사용 경험 축적 후 범용화해야 올바른 추상화 가능

**후속**: 규칙을 config 파일로 분리하여 다른 프로젝트에서도 사용 가능하게 확장.

---

## AD-2: 프롬프트 기반 구현

**결정**: Python/JS 훅보다 SKILL.md 프롬프트로 먼저 구현한다. 프로그래매틱 검증은 arch-check에서만 bash 병행.

**근거**:
- Claude Code 스킬 시스템이 프롬프트 기반으로 동작
- 프롬프트 기반이 빠르게 이터레이션 가능
- 복잡한 판단(설계 적합성 리뷰)은 LLM이 프롬프트로 더 잘 수행
- arch-check처럼 패턴 매칭이 핵심인 경우만 bash/grep 병행

**트레이드오프**: 프롬프트 기반은 결정적(deterministic)이지 않음. 중요한 검증은 bash 스크립트로 보완.

---

## AD-3: `_shared/arix-rules.md`가 Single Source of Truth

**결정**: 설계 문서 전체를 매번 읽지 않고, 강제 가능한 규칙만 추출하여 단일 참조 문서로 통합.

**근거**:
- 설계 문서 4개 합계 ~1,800줄 — 매번 전체 로드는 컨텍스트 낭비
- 스킬마다 필요한 규칙이 다르지만 출처는 같음
- 규칙 변경 시 한 곳만 수정하면 모든 스킬에 반영

**유지보수 규칙**: 설계 문서가 업데이트되면 `arix-rules.md`도 반드시 동기화.

---

## AD-4: Hooks는 최소

**결정**: PreToolUse에서 경량 경고만 제공. 차단(block)하지 않음.

**근거**:
- 훅이 차단하면 개발 흐름이 끊김
- 정보 제공(레이어 표시)만으로도 인지 효과 충분
- 강한 검증은 `/arch-check` 스킬로 명시적 실행
- SessionStart 훅은 향후 arix-rules.md 요약 주입에 사용 예정

---

## AD-5: Contracts-first 원칙 강제

**결정**: `/scaffold` 스킬에서 해당 레이어의 Contracts 프로젝트가 없으면 구현 프로젝트 생성을 차단.

**근거**:
- `docs/repo-design-v1.md` §11 구현 우선순위: Contracts가 1순위
- Contracts 없이 구현하면 인터페이스가 구현에 종속됨
- 교차 레이어 참조는 반드시 Contracts를 통해야 함 (arix-rules.md §3)

---

## AD-6: 점수 체계 (arch-reviewer)

**결정**: 5개 영역 가중 평균으로 A/B/C/D 등급 산출.

| 영역 | 가중치 |
|------|--------|
| 레이어 경계 준수 | 30% |
| Contracts-first 준수 | 20% |
| 4자 역할 분담 | 20% |
| 기술 스택 준수 | 15% |
| 금지 패턴 부재 | 15% |

**근거**:
- 레이어 경계가 가장 중요 (아키텍처의 근본)
- 기술 스택과 금지 패턴은 상대적으로 발견/수정이 쉬움
- 등급은 PR 머지 가이드라인으로 사용 가능 (A/B: 진행, C: 리팩토링, D: 재검토)
